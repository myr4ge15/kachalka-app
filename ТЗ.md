# Техническое задание: веб-приложение для учёта тренировок

**Версия:** 0.1 (черновик для согласования)
**Дата:** 23.06.2026
**Тип:** PWA (веб-приложение, устанавливаемое на телефон) + внешняя БД

---

## 1. Цель

Простое приложение, в котором группа друзей (≈5 человек) отмечает результаты силовых тренировок (упражнение → подходы → вес × повторы) и смотрит динамику весов во времени — прогресс или регресс. Дополнительно — лёгкий соревновательный режим (рейтинг, рекорды).

Ключевые принципы:
- **Скорость ввода** в зале с телефона важнее всего.
- **Бесплатность** на старте.
- **Минимум рутины**: фиксированный список упражнений, шаблоны тренировок.

---

## 2. Пользователи и доступ

- Круг пользователей: ~5 человек, регистрация может быть открытой (но по сути закрытое сообщество).
- **Авторизация по PIN-коду.** На входе пользователь выбирает себя из списка и вводит свой PIN.
- **Админская учётка** — отдельный пользователь с расширенными правами:
  - управление справочником упражнений (добавление/правка/удаление общих упражнений);
  - управление пользователями (создать/удалить, сбросить PIN);
  - модерация данных при необходимости.
- Все обычные пользователи равны в правах.
- **Видимость:** результаты видны всем участникам (общая лента + страницы друг друга), это нужно для рейтинга и сравнения.

---

## 3. Модель данных

### 3.1 Пользователь (`user`)
- id
- имя / никнейм
- PIN (хранится в виде хэша, не в открытом виде)
- роль: `admin` | `member`
- вес тела (текущий, с историей изменений — см. ниже)

### 3.2 Упражнение (`exercise`) — справочник
- id
- название (например, «Жим лёжа»)
- группа / тег: `грудь`, `спина`, `ноги`, `бицепс`, `трицепс`, `плечи` и т.д. (можно несколько тегов)
- признак: общее (из справочника) или пользовательское (добавлено кем-то)
- единица измерения веса: кг (по умолчанию)

> Справочник общий и заранее заполнен типовыми упражнениями. Любой пользователь может **добавить новое упражнение**, если его нет в списке. Чтобы не плодить дубли — при добавлении показывать похожие по названию.

### 3.3 Тренировка (`workout`)
- id
- пользователь
- дата/время
- (необязательно) название/тег дня: «грудь + бицепс», «спина + трицепс» — для фильтра
- список выполненных упражнений

### 3.4 Выполненное упражнение (`workout_exercise`)
- ссылка на тренировку
- ссылка на упражнение из справочника
- список подходов

### 3.5 Подход (`set`)
- номер подхода
- вес (кг)
- количество повторений

> В MVP фиксируем только **вес и повторы**. Поля вроде RPE, времени отдыха, заметок — на будущее.

### 3.6 Вес тела (`bodyweight_log`)
- пользователь, дата, вес — чтобы строить график веса тела отдельно.

### 3.7 Шаблон тренировки (`template`)
- пользователь (или общий)
- название («День груди»)
- список упражнений по умолчанию
- При создании новой тренировки можно выбрать шаблон — упражнения подтянутся автоматически, останется вписать веса/повторы.

---

## 4. Функциональность

### 4.1 Ввод тренировки (главный сценарий, с телефона)
1. Кнопка «Новая тренировка».
2. Выбор шаблона (опционально) или ручное добавление упражнений.
3. Добавление упражнения через **фиксированный список с поиском/фильтром** по названию и группе; если нужного нет — «+ добавить новое».
4. Для каждого упражнения — добавление подходов (вес × повторы). Удобный ввод: крупные кнопки +/−, повтор предыдущего подхода в один тап.
5. Сохранение. Должно работать и **офлайн** (см. раздел 6).

### 4.2 Просмотр прогресса
- **График по конкретному упражнению** во времени. На выбор метрика:
  - рабочий вес (например, лучший/средний рабочий подход за тренировку);
  - расчётный максимум «на раз» (1ПМ, по формуле Эпли/Бжицки на основе вес×повторы);
  - суммарный тоннаж за тренировку (вес × повторы, сумма).
- **Рекорды (PR):** автоматическое выделение нового личного рекорда по упражнению (макс. вес, макс. 1ПМ).
- **График веса тела.**
- **Периоды:** неделя / месяц / произвольный диапазон дат.
- Визуальная подсветка прогресса/регресса (рост — зелёным, спад — красным).

### 4.3 Социальное / соревновательное
- Общая лента последних тренировок друзей.
- **Лидерборд / рейтинг** — например, по 1ПМ в ключевых упражнениях (жим, присед, тяга) или по числу тренировок за период.
- Отметки новых рекордов в ленте.

### 4.4 Справочник упражнений
- Просмотр, поиск, фильтр по группам.
- Добавление пользовательского упражнения.
- Админ управляет общим справочником.

### 4.5 Уведомления (опционально, фаза 2)
GitHub Pages — статичный хостинг и сам по себе уведомления слать не может. Реалистичные варианты:
- **MVP:** внутренние бейджи в приложении («у тебя новый рекорд!», «друг побил твой рекорд»).
- **Фаза 2 (бесплатно):** Telegram-бот + плановая задача (cron) на стороне Supabase Edge Functions или GitHub Actions, которая раз в день шлёт сводку/поздравление с рекордом в общий чат.
- Web Push (системные пуши на телефон) технически возможен через PWA, но требует отдельного сервера отправки — на старте избыточно.

## 5. Архитектура

Так как готового сервера и статических ресурсов нет, а бюджет — ноль, оптимальная связка:

```
┌─────────────────────────────┐
│   Телефон / браузер         │
│   PWA (устанавливается как  │
│   иконка на рабочий стол)   │
│                             │
│  - UI (React или Vue)       │
│  - Service Worker (офлайн)  │
│  - IndexedDB (локальный кэш)│
└──────────────┬──────────────┘
               │ HTTPS (REST / Realtime)
               ▼
┌─────────────────────────────┐
│   Supabase (бесплатный tier)│
│  - PostgreSQL (данные)      │
│  - Auto REST / Realtime API │
│  - Auth + RLS (права)       │
│  - Edge Functions (cron,    │
│    Telegram — фаза 2)       │
└─────────────────────────────┘

Хостинг кода фронтенда: GitHub Pages
Сборка/деплой: GitHub Actions (CI/CD)
```

### 5.1 Почему так

- **GitHub Pages** — бесплатный хостинг статичного сайта (HTML/JS/CSS). Сюда выкладывается собранное приложение.
- **GitHub Actions** — при пуше в репозиторий автоматически собирает фронтенд и публикует на Pages. Именно эта автоматизация и имелась в виду под «через GitHub Actions». Сервер Actions не держит — он только собирает и деплоит.
- **Supabase** — бесплатный облачный PostgreSQL с автоматическим REST/Realtime API и встроенной авторизацией. **Разворачивать ничего не нужно** — это управляемый сервис: регистрируешься, создаёшь проект, получаешь URL и ключ, обращаешься к БД прямо из браузера по HTTPS.
- **Одна БД для всех клиентов.** И сайт на Pages, и «приложение» на телефоне — это один и тот же веб-клиент. Оба обращаются к одной Supabase. То есть БД «привязана» и к мобилке, и к Pages автоматически — отдельной мобильной БД не нужно.

### 5.2 PWA и офлайн-синхронизация

- Приложение оформляется как **PWA**: добавляется на телефон иконкой, открывается на весь экран, работает как нативное.
- **Офлайн:** записи тренировки сохраняются локально в IndexedDB. При появлении сети — фоновая синхронизация отправляет их в Supabase. Конфликты маловероятны (каждый пишет свои данные), разрешаются по времени.

### 5.3 Авторизация по PIN

- Простой вариант: таблица пользователей, PIN хранится хэшем. Вход = выбор пользователя + ввод PIN.
- Права на чтение/запись разграничиваются политиками **RLS** в Supabase (свои данные — пишу, чужие — только читаю; админ — больше).

### 5.4 Лимиты бесплатного tier (ориентир)

- Supabase free: ~500 МБ БД, достаточно для текстовых записей тренировок на годы вперёд для 5 человек.
- GitHub Pages / Actions free: с запасом хватает для такого проекта.
- Риск: Supabase может «усыплять» неактивный проект — лечится лёгким плановым пингом.

## 6. Нефункциональные требования

- **Платформа:** в первую очередь мобильный браузер (телефон). Десктоп — как бонус, тот же сайт.
- **Язык интерфейса:** русский.
- **Единицы:** килограммы.
- **Офлайн-режим** с последующей синхронизацией.
- **Дизайн:** баланс между удобством/скоростью ввода и приятным видом. Мобильный-first, крупные элементы под палец, минимум экранов до сохранения подхода.
- **Бесплатно** на старте; архитектура допускает переход на платные tier без переписывания.

## 7. Объём MVP vs. дальнейшее развитие

### MVP (первая версия)
- PIN-вход, список пользователей, админ.
- Справочник упражнений (предзаполненный) + добавление своих.
- Ввод тренировки: упражнения → подходы (вес × повторы), офлайн.
- График прогресса по упражнению (рабочий вес + 1ПМ) с периодами неделя/месяц/произвольно.
- Авто-рекорды (PR).
- Общая лента тренировок.

### Фаза 2
- Шаблоны тренировок.
- Лидерборд / рейтинг.
- График веса тела.
- Теги дней (грудь+бицепс и т.д.) и фильтрация по ним.
- Уведомления через Telegram-бота.

## 8. Согласованные решения

1. **Фронтенд — React.**
2. **PIN — 4 цифры.**
3. **Лидерборд — только по жиму лёжа** (1ПМ). Остальные упражнения — без рейтинга.
4. **График веса тела — не в MVP**, перенесён в фазу 2 (учёт веса тела как данных остаётся, визуализация — позже).
5. **Справочник упражнений — предзаполнен** стартовым списком (см. Приложение A) с возможностью добавлять свои.

## Приложение A. Стартовый справочник упражнений

> Базовый набор, загружается в БД при инициализации. Любой пользователь может дополнить.

### Грудь
- Жим лёжа со штангой *(используется для лидерборда)*
- Жим гантелей лёжа
- Жим лёжа на наклонной скамье
- Жим в тренажёре (хаммер)
- Разведение гантелей лёжа
- Сведения в кроссовере / бабочка
- Отжимания на брусьях (грудь)

### Спина
- Становая тяга
- Подтягивания
- Тяга верхнего блока
- Тяга штанги в наклоне
- Тяга гантели одной рукой
- Горизонтальная тяга блока
- Гиперэкстензия

### Ноги
- Приседания со штангой
- Жим ногами
- Разгибание ног в тренажёре
- Сгибание ног в тренажёре
- Выпады с гантелями
- Подъём на носки (икры)
- Румынская тяга

### Плечи
- Жим штанги стоя
- Жим гантелей сидя
- Махи гантелями в стороны
- Махи в наклоне (задняя дельта)
- Тяга штанги к подбородку

### Бицепс
- Подъём штанги на бицепс
- Подъём гантелей на бицепс
- Молотковые сгибания
- Сгибания на скамье Скотта

### Трицепс
- Жим узким хватом
- Разгибание на блоке
- Французский жим
- Отжимания на брусьях (трицепс)

### Пресс / кор
- Скручивания
- Подъём ног в висе
- Планка

Макет основных экранов — в файле `concept.html` (открывается в браузере). Показаны четыре экрана: вход по PIN, ввод тренировки (с шаблоном и быстрым повтором подхода), прогресс (график максимума в жиме с подсветкой роста/рекорда), лидерборд + лента. ## Приложение C. Стек и старт разработки

### Технологии
- **Фронтенд:** React + Vite, оформление как PWA (манифест + Service Worker).
- **Локальное хранилище / офлайн:** IndexedDB (через `idb` или `Dexie`), очередь несинхронизированных записей.
- **БД и API:** Supabase (PostgreSQL + auto REST/Realtime + Auth + RLS).
- **Графики:** легковесная библиотека (Recharts или Chart.js).
- **Хостинг:** GitHub Pages.
- **CI/CD:** GitHub Actions — сборка и деплой на Pages при пуше в `main`.

### Примерная структура репозитория
```
/
├─ .github/workflows/deploy.yml   # сборка + публикация на Pages
├─ public/                        # манифест PWA, иконки
├─ src/
│  ├─ components/                 # UI-компоненты
│  ├─ screens/                    # экраны (вход, тренировка, прогресс, лидерборд)
│  ├─ db/                         # клиент Supabase + IndexedDB, синхронизация
│  ├─ lib/                        # расчёт 1ПМ, формулы, утилиты
│  └─ App.jsx
├─ supabase/
│  ├─ schema.sql                  # таблицы + RLS-политики
│  └─ seed.sql                    # стартовый справочник упражнений (Приложение A)
├─ vite.config.js
└─ package.json
```

### Подготовка перед стартом (вне кода)
1. Завести проект в Supabase, получить URL и anon-ключ.
2. Создать GitHub-репозиторий, включить Pages (источник — GitHub Actions).
3. Прописать ключи Supabase в секреты репозитория / переменные окружения сборки.

### Что считаем «готовым MVP»
Пользователь входит по PIN, заводит тренировку (офлайн в том числе), данные синхронизируются в Supabase, виден график прогресса по жиму и лидерборд по жиму, рекорды отмечаются автоматически.
